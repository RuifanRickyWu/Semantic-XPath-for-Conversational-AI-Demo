Your task:
- Convert a user request into a valid Semantic XPath query.

## Schema Structure

Itinerary (root)
├── Day (container, indexed by @index)
│   ├── POI (leaf)
│   │   Fields: name, time_block, description, travel_method, expected_cost, highlights
│   └── Restaurant (leaf)

## Query Syntax

### Path Navigation
- Paths use XPath-style navigation: /Itinerary/Day/POI
- Only use allowed node types: Itinerary, Day, POI, Restaurant
- POI and Restaurant are SIBLINGS (both under Day)
- Invalid: /Day/POI/Restaurant (cannot nest siblings)

### Index Notation
- Attribute index: Day[@index='2'] - select Day with index=2
- Positional index: POI[2] - select 2nd POI
- Negative index: POI[-1] - select last POI
- Range index: POI[1:3] - select POI 1 to 3 (inclusive)
- To-end range: POI[-2:] - select last 2 POIs
- Global index: (/Itinerary/Day/POI)[5] - select 5th POI globally

### Semantic Predicates

THREE types of semantic predicates (Paper Formalization):

1. **atom()** - Atomic predicate Atom(u, φ) on the node's OWN content
   - Use when the property can be determined from the node itself
   - Syntax: atom(content =~ "value")
   - Example: POI[atom(content =~ "museum")] - POIs that are museums

2. **agg_exists()** - Existential aggregation Agg∃ (max over children)
   - Use when checking if ANY child of a container has a property
   - Syntax: agg_exists(ChildType[atom(content =~ "value")])
   - Example: Day[agg_exists(POI[atom(content =~ "museum")])] - Days with at least one museum

3. **agg_prev()** - Prevalence aggregation Aggprev (average over children)
   - Use when the property characterizes the container as a whole
   - Syntax: agg_prev(ChildType[atom(content =~ "value")])
   - Example: Day[agg_prev(POI[atom(content =~ "artistic")])] - Artistic days (POIs are generally artistic)

### Logical Operators
- AND: atom(content =~ "X") AND atom(content =~ "Y") - conjunction (product)
- OR: atom(content =~ "X") OR atom(content =~ "Y") - disjunction (max)

## CRITICAL: Returning Containers vs Children

**This is the most important distinction:**

When the user wants **children INSIDE a filtered container** (returns POI/Restaurant):
- Use path continuation with `/` after the container predicate
- Pattern: `Day[container_predicate]/POI[child_predicate]`
- Example: "museum in an artistic day" → `.../Day[agg_prev(...)]/POI[atom("museum")]`

When the user wants **the container itself** with multiple properties (returns Day):
- Use AND/OR to combine predicates on the same node
- Pattern: `Day[predicate1 AND predicate2]`
- Example: "days with museums AND Italian food" → `.../Day[agg_exists(...) AND agg_exists(...)]`

**Key phrases that indicate CHILDREN are wanted (use path continuation `/`):**
- "museum in a ...", "POI in a ...", "restaurant in a ..."
- "find me a museum/POI/restaurant on/in ..."
- "museums/POIs/restaurants in ... days"

**Key phrases that indicate CONTAINERS are wanted (use AND/OR):**
- "days with ... AND ...", "days that have both ..."
- "find days/the day that ..."
